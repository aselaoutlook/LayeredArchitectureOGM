Naming Projects & Namespaces

Framework has to be renamed to align with the project you work. Below steps brief what you need to carry out before start actual coding for client.

Note


ReSharper tool can perform much of the below steps in few clicks

Jet Brains claims

  1. Change solution name to FCL.Cockerham.Ogsm. Ex:- FCL.Web.Framework = FCL.Client.Project Go to properties in solutions to change
  2. Change namespaces of each project that is named as FCL.Cockerham.Ogsm.xxxx Ex:- FCL.Cockerham.Ogsm.Admin = FCL.Microsoft.SharePoint.Admin
    1. Go to properties in project and change project file
    2. Right click project, properties, Application tab change assembly name and default namespace Keep Client.Project part simple and not too long.
  3. Change directory names of each project to match with above guidelines. Ex:- After you change FCL.Web.Framework.Admin project as in step 2
    1. remove the project from solution and rename the FCL.Web.Framework.Admin directory to FCL.Cockerham.Ogsm.Admin
    2. Re add the project to solution from the renamed directory
  4. Namespaces in class files must be updated to match with the above changes
  5. Update AssemblyInfo (in AssemblyInfo.cs) details as necessary
Attention

If you are adding any functionality that can be included into the framework (not client specific) make sure you follow project naming and namespace naming guidelines. Ex:- FCL.Web.Framework.XXXXXX

Auditing Framework

Note

Framework has a built in auditing module. This audits every change that happense in the database. Main classes/interface for auditing are as below;

  • FCL.Web.Framework.Core.Common.Contracts\IAuditableEntity.cs - Primary interface which provides a property for entities to control whether auditing should happen or not. By default every entity gets audited.

  • FCL.Cockerham.Ogsm.Data.Contracts\Repository Interfaces\IAuditRepository.cs - Provides methods to interace with audit table. By default it exposes a single method where it returns ALL records in audit table.

  • FCL.Web.Framework.Core.Common\Core\EntityBase.cs - Implements IAuditableEntity so that all entities by default gets AuditChanges property.

  • FCL.Web.Framework\FCL.Cockerham.Ogsm.Entities\Audit.cs - Represents the Audit table in DB

  • FCL.Cockerham.Ogsm.Data\FclDBContext.cs - Contains the logic that performs the actual auditing work. This has overriden the SaveChanges method. This checks whether the incoming entity has auditing enabled (AuditChanges property). If enabled it collects all the data and save those to Audit table.
  • FCL.Cockerham.Ogsm.Admin\Views\Audit\Index.cshtml - Has a sample table to display audited data. You can customize this as you wish.

All audit data are written to Audit table having below structure

AuditId | EventDate | Table | Action | User | RecordID | Old Data | New Data

All the audit related data is written as a JSON string to RecordID, Old Data & New Data fields.

How to use How to use

By default auditing is enabled for every entity. However if you want to disable auditing for a particular entity you have to explicitly disable it by [Entity].AuditChanges = false;

Template Editor Extention

Template editor can be used to allow customers to edit their email templates. This is built to use with Postal where you can create email templates in razor sytax.

This extention uses a customized HTML editor by summernote

Below are the files in use for this extention

  • FCL.Cockerham.Ogsm.Admin\Content\summernote.css - Styles used by the rich text editor.

  • FCL.Cockerham.Ogsm.Admin\Scripts\summernote.js - Customized version of the core summer note editor.

  • FCL.Web.Framework.Core.RazorExtentions\template-editor.js - Implementation of summernote editor at client end.

  • FCL.Web.Framework\FCL.Cockerham.Ogsm.Entities\Audit.cs - Represents the Audit table in DB

  • FCL.Web.Framework.Core.RazorExtentions\TemplateEditorScript.resx - Contains the above script as a resource.
  • FCL.Web.Framework.Core.RazorExtentions\TemplateEditorExtension.cs - HTML Helper class hiding the complexities
  • FCL.Cockerham.Ogsm.Admin\Views\TemplateEditor\Index.cshtml - Example implementation of the template editor.

How to use

@(Html.TemplateEditorFor().EditorId("summernote")
    .EditorClasses("1 2 3") .Height(300)
    .EditorMaxHeight(350)
    .IsTemplateEditor(true)
    .TagRegxp("@View\\w+\\w+")
    .SaveTemplateCallback("console.log(savedData)")
    .ResetTemplateCallback("console.log('reset')")
    .ToHtml());

Above code will render all the scripts required for the editor to work.

Note
 
  • IsTemplateEditor - If set to false, renders a generic rich text editor

  • TagRegXP - A valid javascript regular expression that will be used to parse incoming template data and highlight user tags. For example, if you are using Postal.net razor files for email templates, each placeholder can be specified by @ViewBag.Property. So for you to extract all these variable data and make a place holder, you can provide a regular expression. By default it is set to @View\\w+\\w+ which will find all @ViewBag.Property placeholders and convert them to tags in the richtext editor.
  •  SaveTemplateCallback - This gets called when user clicks the [Save Template] button. You will get the updated template data via the call back.
  • ResetTemplateCallback - Gets called when user clicks the [ResetTemplateCallback] button. You might want to use this to reset the template data.
